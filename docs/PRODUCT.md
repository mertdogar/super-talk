# Product

## Register

brand

## Users

Developers evaluating or adopting super-talk. They've landed here from a
README, a link, or a search because they want to understand what
super-talk is and decide whether to run it. Two moments matter: the
first-impression skim (what is this, why should I care, is it serious)
and the get-it-running pass (copy the command, start a hub, connect an
agent). They are technical and impatient with fluff; they reward a clear
point of view and a fast path to a working hub.

## Product Purpose

The super-talk documentation site (VitePress). Its job is to convert a
visitor into someone running a hub: state what super-talk is and why it's
different, then get them from zero to agents-talking quickly. It is also
the canonical reference for the hub, the plugin, and the web UI. Success
is a reader who finishes the landing with a clear mental model and runs
the getting-started commands without friction.

## Brand Personality

Bold and opinionated. The site has a point of view about how AI agents
should communicate — one hub, shared channels, messages pushed straight
into each agent — and it states that confidently rather than hedging.
Strong hero, decisive voice, memorable; not neutral reference-doc
wallpaper. It still reads as **precise and technical** underneath: the
confidence is earned by clarity, not by marketing gloss.

This site shares the product's identity. Docs and the web app are one
brand — same palette, type, and voice — so moving between them feels like
one product. The docs surface may be more expressive than the working app
UI, but never a different brand.

## Anti-references

- **Neutral template docs.** Default VitePress with no point of view —
  generic sidebar, stock hero, interchangeable with any other project's
  docs. The site should be unmistakably super-talk.
- **Marketing-hype landing.** No vague enterprise buzzwords, gradient
  hero metrics, or "revolutionary platform" copy. Boldness comes from a
  clear thesis, not adjectives.
- **A Slack-clone look or a generic SaaS-purple gradient aesthetic**,
  inherited from the app's anti-references — the shared brand carries
  these forward.

## Design Principles

- **Lead with the thesis.** The landing makes the argument for super-talk
  in the first screen — what it is and why one-hub shared channels beat
  the alternatives — before any feature tour.
- **Fast path to a running hub.** Getting-started is never more than a
  scroll or a click away; copyable commands, minimal ceremony, momentum
  toward a working setup.
- **One brand across surfaces.** Reuse the app's palette, type, and voice
  so docs and product reinforce each other.
- **Confidence through clarity.** The opinionated voice is backed by
  precise, accurate, well-structured content — never hype standing in for
  substance.
- **Show the system honestly.** Explain the real architecture and its
  limits plainly; credibility with a technical audience comes from not
  overselling.

## Accessibility & Inclusion

WCAG AA. Body and prose text ≥4.5:1 contrast (large text ≥3:1), full
keyboard navigation, visible focus states, and `prefers-reduced-motion`
honored on every animation (crossfade or instant fallback). Code blocks
and callouts must not rely on color alone to convey meaning.
